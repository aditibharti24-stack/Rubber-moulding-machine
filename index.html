Publishes to /machine/* topics, buffers on disconnect, and reconnects automatically.

Why MQTT

Lightweight publish-subscribe behavior suits high-frequency machine telemetry better than polling-driven HTTP.

Fault Tolerance

Client buffering, retries, acknowledgments, and QoS 2 protect against message loss.

Security IEC 62443

TLS, authenticated broker access, and secure transport align with industrial cybersecurity practices.

Client

PLC publishes temperature, pressure, and vibration topics and caches payloads locally during outages.

Server

Broker and central server subscribe, validate, store, and expose records to the analytics layer.

Protocol Comparison

MQTT: lightweight pub/sub for live telemetry and alerts OPC-UA: richer semantics but heavier for this compact IIoT prototype Modbus: simple field exchange, weaker for cloud-style event routing

Integrity And Fault Tolerance

Sequence ordering and acknowledgments protect message continuity Client buffering supports reconnect and delayed broker delivery Checksums, retries, and QoS 2 reduce duplication or packet loss

LAN, WAN, And HMI Scope

Industrial LAN handles machine-to-server transport at plant level WAN/cloud links extend reporting, dashboards, and remote supervision HMI, alert beacons, and mobile web clients consume the same server data

Weak Entity Logic

Conceptually, SensorReading behaves like a weak entity because its business meaning depends on its owner Sensor plus observation time. The SQL design still uses reading_id as a surrogate primary key for fast indexing and simpler joins.

Relationship And Key Logic

Primary keys uniquely identify machines, sensors, batches, alerts, and technicians Foreign keys enforce referential integrity between machine, sensor, maintenance, and alert tables Views, triggers, and procedures support both real-time monitoring and historical analysis

-- Trigger inserts an alert when a reading crosses the threshold
CREATE TRIGGER auto_alert_on_threshold
AFTER INSERT ON SensorReading
FOR EACH ROW
BEGIN
  IF NEW.value > 200 THEN
    INSERT INTO Alert(sensor_id, triggered_at, alert_type, threshold_value, actual_value, resolved)
    VALUES (NEW.sensor_id, NEW.timestamp, 'HIGH_TEMP', 200, NEW.value, FALSE);
  END IF;
END;

CNDC To DBMS To Statistics Decision Pipeline

The complete prototype follows the required workflow: define the industrial problem, move machine data through secure networking, store it in a relational database, analyze it with inferential statistics, and turn the results into maintenance and quality decisions.

1. Business Boundary

Sensors monitor heat, pressure, vibration, machine health, and batch quality for a 4-pillar compression molding machine.

2. CNDC Flow

Controller to broker to server communication uses MQTT over TCP/IP with fault tolerance, QoS, TLS, and plant LAN routing.

3. DBMS Layer

Relational tables, foreign keys, views, indexes, triggers, and procedures support both live dashboards and historical analysis.

4. Statistical Decision

Hypothesis tests, ANOVA, regression, control charts, anomaly flags, and reliability curves drive process optimization.
