SmartMold IoT Project Overview

This overview now explains what the SmartMold product is, why it matters for a 4-pillar rubber molding machine, and how the full implementation moves from sensors to decisions.

Product Snapshot

What SmartMold IoT Delivers

SmartMold IoT converts a conventional rubber molding machine into a connected industrial product that senses process temperature, hydraulic pressure, and mechanical vibration in real time, stores machine history, and supports better production, maintenance, and quality decisions.

Real-time telemetry visibility for operators and supervisors Predictive fault awareness before breakdown and downtime Historical storage for short-term and long-term trend comparison Actionable dashboards for process control, health review, and reporting
Implementation Path

How We Will Implement The Product

01
Instrument the machine

Install temperature, pressure, and vibration sensors on the heater zone, hydraulic line, and machine frame.

02
Connect the controller

Interface sensors with a PLC or ESP32 so readings are sampled, validated, and prepared for transmission.

03
Send data through MQTT

Publish secure MQTT messages over the Industrial LAN to move live machine data to the broker and server.

04
Store and organize history

Write timestamped readings, alerts, maintenance records, and production quality data into MySQL for comparison and reporting.

05
Visualize and decide

Use the dashboard, SQL insights, and statistics layer to detect faults, compare trends, and improve reliability.

The Problem

Traditional 4-pillar rubber molding machines have no real-time monitoring, no predictive maintenance, and no data-driven control, causing poor quality, energy waste, and unplanned downtime.

The Solution

SmartMold IoT deploys temperature, pressure, and vibration sensors on the machine and sends telemetry by MQTT over TCP/IP through an Industrial LAN to a central server and MySQL database for visualization, predictive maintenance, and statistical process control.

-- Trigger inserts an alert when a reading crosses the threshold
CREATE TRIGGER auto_alert_on_threshold
AFTER INSERT ON SensorReading
FOR EACH ROW
BEGIN
  IF NEW.value > 200 THEN
    INSERT INTO Alert(sensor_id, triggered_at, alert_type, threshold_value, actual_value, resolved)
    VALUES (NEW.sensor_id, NEW.timestamp, 'HIGH_TEMP', 200, NEW.value, FALSE);
  END IF;
END;

VIEWS / TRIGGER / PROCEDURE

Database Automation Layer

View, trigger, and stored procedure support live status, automatic alerts, and maintenance due checks.

-- View
CREATE VIEW live_sensor_status AS
SELECT s.sensor_type, r.value, r.status, r.timestamp
FROM Sensor s JOIN SensorReading r ON s.sensor_id = r.sensor_id
WHERE r.timestamp = (SELECT MAX(timestamp) FROM SensorReading);
-- Trigger
CREATE TRIGGER auto_alert AFTER INSERT ON SensorReading
FOR EACH ROW
BEGIN
  IF NEW.value > 200 THEN
    INSERT INTO Alert(sensor_id, triggered_at, alert_type, actual_value, resolved)
    VALUES (NEW.sensor_id, NOW(), 'CRITICAL_TEMP', NEW.value, FALSE);
  END IF;
END;
-- Stored Procedure
CREATE PROCEDURE get_maintenance_due()
BEGIN
  SELECT machine_id, MAX(date) AS last_maintenance
  FROM MaintenanceRecord
  GROUP BY machine_id
  HAVING DATEDIFF(NOW(), MAX(date)) > 30;
END;

DBMS — 1.5 marks: Normalisation to highest suitable form

Normalization Compliance

The SmartMold relational schema is normalized to remove redundancy and preserve data integrity while still remaining practical for monitoring and analytics.

What Is Normalization?

Normalization is the process of organizing relational tables to reduce redundancy, remove update anomalies, and improve consistency through proper functional dependency handling.

Functional Dependencies

TableFunctional Dependency
SensorReadingreading_id → sensor_id, value, timestamp, status
Sensorsensor_id → machine_id, sensor_type, unit
Machinemachine_id → name, location, model, installation_date
MaintenanceRecordmaintenance_id → machine_id, technician_id, date, cost
ProductionBatchbatch_id → machine_id, quality_score, defect_count
Alertalert_id → sensor_id, triggered_at, threshold, actual

Normal Forms Applied To SmartMold

1NF

First Normal Form

  • All values are atomic.
  • No repeating groups exist.

Compliant: one reading value and one status are stored in each row.

2NF

Second Normal Form

  • Already in 1NF.
  • No partial dependency on composite keys.

Compliant: SmartMold uses single-column surrogate primary keys, so partial dependency cannot occur.

3NF

Third Normal Form

  • No transitive dependency.
  • Non-key attributes depend directly on the primary key.

Compliant: technician details and sensor metadata are stored in their own tables.

BCNF

BCNF

  • Every determinant is a super key.
  • Stronger form of 3NF.

Compliant: each core SmartMold table uses a single candidate key.

4NF

Fourth Normal Form

  • No unwanted multi-valued dependency.
  • Independent facts stay in separate tables.

Compliant: sensor types and maintenance types are not mixed into one multi-valued relation.

5NF

Fifth Normal Form

  • No join dependency beyond keys.
  • Decomposition remains lossless.

Compliant: Machine, Sensor, and SensorReading rejoin cleanly through foreign keys.
